# Migration Validation: Extend Venue Users for Employee Management

**Migration File:** `20240101000024_extend_venue_users_for_employees.sql`

**Requirements:** 6.1, 6.2, 6.4, 6.5

## Purpose

This migration extends the `venue_users` table to support employee invitation workflow and role-based access control for venue staff members.

## Changes Made

### 1. Added Invitation Tracking Columns

- `invited_by` (UUID, nullable): References the administrator who invited the employee
- `invited_at` (TIMESTAMPTZ): Timestamp when invitation was sent (defaults to NOW())
- `accepted_at` (TIMESTAMPTZ, nullable): Timestamp when employee accepted invitation
- `deactivated_at` (TIMESTAMPTZ, nullable): Timestamp when employee account was deactivated

### 2. Updated Role Constraint

- Dropped existing `venue_users_role_check` constraint
- Added new constraint supporting three roles: `administrator`, `editor`, `viewer`
- Previous role values may need migration if they don't match new constraint

### 3. Added Indexes

- `idx_venue_users_active`: Optimizes queries for active employees (WHERE deactivated_at IS NULL)
- `idx_venue_users_pending_invitations`: Optimizes queries for pending invitations

### 4. Added Documentation

- Column comments explaining purpose of each new field
- Constraint comment documenting valid role values

## Data Migration Considerations

### Existing Role Values

If the `venue_users` table has existing rows with role values other than 'administrator', 'editor', or 'viewer', they must be migrated before applying this migration:

```sql

-- Example migration (adjust as needed):
```

### Default Values for New Columns

- `invited_at`: Will be set to NOW() for existing rows
- `invited_by`, `accepted_at`, `deactivated_at`: Will be NULL for existing rows
- Consider setting `accepted_at = created_at` for existing employees to indicate they're already active

```sql
-- Optional: Mark existing employees as accepted
UPDATE venue_users
SET accepted_at = created_at
WHERE accepted_at IS NULL;
```
