# Migration Validation: 20240101000027_create_venue_analytics_view.sql

## Purpose
Create materialized view for venue analytics to improve dashboard query performance.

## Requirements Addressed
- **Requirement 20.1**: Display total booking count, revenue trends, and performance metrics
- **Requirement 20.2**: Display profile views and booking conversion rate

## Changes Made

### 1. Materialized View: venue_analytics_summary
Pre-computes venue analytics metrics including:
- Total bookings (all statuses)
- Completed bookings count
- Cancelled bookings count
- Total revenue (from confirmed/completed bookings)
- Average group size
- Current rating and review count
- Profile views (from search history)
- Booking conversion rate (bookings / profile views)

### 2. Indexes
- `idx_venue_analytics_venue`: Unique index on venue_id for fast lookups
- `idx_venue_analytics_bookings`: Index for sorting by booking count
- `idx_venue_analytics_revenue`: Index for sorting by revenue

### 3. Refresh Function
- `refresh_venue_analytics()`: Function to refresh the materialized view concurrently

## Performance Impact
- **Positive**: Dashboard queries will be much faster (pre-computed aggregations)
- **Storage**: Minimal (one row per venue)
- **Refresh Cost**: Moderate (should be run periodically, e.g., hourly or daily)

## Notes
- The materialized view should be refreshed periodically (e.g., via cron job or scheduled task)
- Use `REFRESH MATERIALIZED VIEW CONCURRENTLY` to avoid locking
- The view depends on `venue_bookings` and `venue_search_history` tables
- If `venue_bookings` table doesn't exist yet, the view will still be created but with NULL values
