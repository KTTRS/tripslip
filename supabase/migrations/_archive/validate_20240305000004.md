# Migration Validation: 20240305000004_create_logging_tables.sql

## Overview
This migration enhances the existing logging infrastructure by adding comprehensive cleanup functions and monitoring capabilities for rate_limits, email_logs, and sms_logs tables.

## Prerequisites
- Migration 20240101000011 (rate_limits table)
- Migration 20240101000045 (email_logs and sms_logs tables)

## Changes Made

### 1. Cleanup Functions
- ✅ `cleanup_old_rate_limits()` - Removes rate limit records older than 1 hour
- ✅ `cleanup_old_email_logs()` - Removes email logs older than 90 days
- ✅ `cleanup_old_sms_logs()` - Removes SMS logs older than 90 days
- ✅ `cleanup_all_logs()` - Master function to run all cleanup jobs

### 2. Performance Indexes
- ✅ `idx_rate_limits_cleanup` - Partial index for cleanup queries
- ✅ `idx_email_logs_cleanup` - Partial index for cleanup queries
- ✅ `idx_sms_logs_cleanup` - Partial index for cleanup queries

### 3. Monitoring Functions
- ✅ `get_logging_statistics()` - Returns comprehensive statistics about all logging tables

## Performance Considerations

### Index Usage
The partial indexes created will improve cleanup query performance:
- Only index rows that match the cleanup criteria
- Reduces index size and maintenance overhead
- Speeds up DELETE operations during cleanup

### Cleanup Frequency Recommendations
- **rate_limits**: Run every hour (or more frequently if high traffic)
- **email_logs**: Run daily (90-day retention is sufficient for audit)
- **sms_logs**: Run daily (90-day retention is sufficient for audit)

### Storage Estimates
Assuming moderate usage:
- **rate_limits**: ~1000 records/hour = 24KB/hour (auto-cleaned)
- **email_logs**: ~100 emails/day = 365KB/90 days = ~33MB
- **sms_logs**: ~50 SMS/day = 183KB/90 days = ~16MB

Total storage for logs: ~50MB (negligible)

## Production Deployment

### Scheduling Cleanup Jobs

Choose one of these methods to schedule cleanup jobs in production:

#### Option 1: Supabase Edge Function (Recommended)
Create an Edge Function that calls the cleanup function and schedule it with GitHub Actions or external cron:

```typescript
// supabase/functions/cleanup-logs/index.ts
import { createClient } from '@supabase/supabase-js';

Deno.serve(async (req) => {
  const supabase = createClient(
    Deno.env.get('SUPABASE_URL')!,
    Deno.env.get('SUPABASE_SERVICE_ROLE_KEY')!
  );
  
  const { data, error } = await supabase.rpc('cleanup_all_logs');
  
  if (error) {
    return new Response(JSON.stringify({ error }), { status: 500 });
  }
  
  return new Response(JSON.stringify({ success: true, data }), { status: 200 });
});
```

#### Option 2: GitHub Actions Workflow
```yaml
# .github/workflows/cleanup-logs.yml
name: Cleanup Logs
on:
  schedule:
    - cron: '0 2 * * *' # Daily at 2 AM UTC
  workflow_dispatch: # Allow manual trigger

jobs:
  cleanup:
    runs-on: ubuntu-latest
    steps:
      - name: Cleanup logs
        run: |
          curl -X POST \
            -H "Authorization: Bearer ${{ secrets.SUPABASE_SERVICE_ROLE_KEY }}" \
            -H "Content-Type: application/json" \
            https://${{ secrets.SUPABASE_PROJECT_REF }}.supabase.co/rest/v1/rpc/cleanup_all_logs
```

#### Option 3: pg_cron Extension (If Available)
```sql
-- Enable pg_cron extension (requires superuser)
CREATE EXTENSION IF NOT EXISTS pg_cron;

-- Schedule daily cleanup at 2 AM
SELECT cron.schedule(
  'cleanup-logs-daily',
  '0 2 * * *',
  'SELECT cleanup_all_logs()'
);

-- View scheduled jobs
SELECT * FROM cron.job;
```

## Monitoring

### Set Up Alerts
Monitor cleanup job execution and log table sizes:

1. **Log table size alerts**: Alert if tables exceed expected size
2. **Cleanup job failures**: Alert if cleanup functions return errors
3. **Statistics monitoring**: Track trends in log volume

### Example Monitoring Query
```sql
-- Check if cleanup is working (run daily)
SELECT 
  table_name,
  total_rows,
  rows_last_24h,
  CASE 
    WHEN table_name = 'rate_limits' AND total_rows > 10000 THEN 'WARNING: rate_limits not being cleaned'
    WHEN table_name IN ('email_logs', 'sms_logs') AND total_rows > 100000 THEN 'WARNING: logs accumulating'
    ELSE 'OK'
  END AS status
FROM get_logging_statistics();
```
